## Project-Specific AI Context
_Last reflected: 2026-04-28 · 10+ sessions analyzed_

### Conventions
- MCP adapter dispatch lives in `src/mcp_adapter.jl`; session lifecycle tools are wired there alongside `mcp_call_tool`
- Hybrid session identity: every session has a UUID (canonical) plus an optional name alias — both are valid handles in all ops
- `create_named_session!` is a **server-side** seed-session helper; the **client interface** for creating sessions is the `new-session` RPC op
- Security validations applied at middleware boundaries: session names validated on entry, eval repr truncated, TCP message size capped, unsupported `module`/`timeout` fields rejected
- CI runs Vale and typos as parallel jobs (split from a single job) — do not merge them back

### Common Gotchas
- Pre-commit `end-of-file-fixer` hook modifies files in place on failure; always re-stage and re-commit after hook auto-fixes
- `wai reflect` runs in agent mode — it prints a context prompt and waits for `wai reflect --inject-content` to complete; it is NOT a one-shot command
- Pipeline steps (implement, review, fix, commit) must run inside subagents (already in bd memories)
- Source changes from a session are NOT automatically committed; check `git status` at session start to catch carry-over uncommitted work
- Handoff files generated by `wai close` are empty templates by default; they will not contain useful context unless explicitly filled in

### Architecture Notes
- Middleware stack (full): SessionMiddleware → EvalMiddleware → DescribeMiddleware → InterruptMiddleware → LoadFileMiddleware → CompletionMiddleware → LookupMiddleware → StdinMiddleware → UnknownOpMiddleware
- Transport: TCP listener, newline-delimited JSON framing, flat envelope; `done` status message always terminates a stream
- `serve_multi` handles concurrent multi-listener orchestration with listener-global session limits
- Graceful shutdown, bounded audit logging, rate limiting, and eval timeout are all wired in as of Phase 8
- `eval-id` and `interrupt-id` fields on eval responses enable per-eval targeting (e.g., interrupt a specific in-flight eval)
- Session status flags: `session-already-exists`, `path-not-allowed` (documented in spec)
- Phase roadmap status: Phases 4A–9 are **complete**. Only open work is a P1 doc bug (`REPLy_jl-dzd`): sessions how-to is missing client RPC op documentation
